COTIZADOR PM/CP - FUNCIONAMIENTO GENERAL DEL CODIGO
Ultima actualizacion: 2026-04-13

Este documento resume la arquitectura vigente del sistema, el mapa real de
archivos y la forma en que se conectan frontend, PocketBase, tenants, seguridad
y documentos.

===============================================================================
1. ESTRUCTURA DEL REPOSITORIO
===============================================================================

- `frontend/`
  HTML, JS, CSS, plantillas PDF, assets y carpeta publica para produccion
- `backend/`
  `pocketbase.exe`, hooks JS, migraciones, bases SQLite, logs
- `development/`
  scripts de arranque local y auditoria smoke
- `production/`
  scripts de servicio Windows y despliegue
- `docs/`
  documentacion tecnica vigente

===============================================================================
2. TENANTS Y RUNTIME
===============================================================================

El sistema opera con dos tenants lógicos:

- Plaza Mayor
  - tenant: `plaza_mayor`
  - schema historico/runtime: `finanzas`
  - frontend administrativo principal: `frontend/client/cotizador/`

- Casa de Piedra
  - tenant: `casa_de_piedra`
  - schema historico/runtime: `finanzas_casadepiedra`
  - frontend administrativo principal: `frontend/client/cotizadorcp/`

La resolucion de tenant ocurre principalmente en:

- `frontend/client/js/hub-config.js`
- `frontend/client/services/pb-core.js`
- pathname del modulo (`/cotizador/` vs `/cotizadorcp/`)

===============================================================================
3. BOOTSTRAP DEL FRONTEND
===============================================================================

Piezas base del runtime:

- `frontend/client/config/hub-runtime.json`

- `frontend/client/js/hub-config.js`

  login, sesion, refresh, logout y sincronizacion entre tabs
- `frontend/client/services/security.js`
  sanitizacion y validaciones reutilizables

===============================================================================
6. DIFERENCIA OPERATIVA PM VS CP
===============================================================================

Plaza Mayor trabaja principalmente con espacios publicitarios.

Casa de Piedra es hibrido:

- categoria `publicidad`
  reutiliza la logica comercial de Plaza Mayor
  maneja permanencia, precio personalizado, impuestos y convenio
- categoria `espacio`
  usa logica de eventos/salones
  maneja personas, horario, premontaje, horas extra y disponibilidad por fecha

Regla critica de CP:

- una misma cotizacion NO puede mezclar `PUBLICIDAD` con `ESPACIO`
- la validacion se aplica en `frontend/client/cotizadorcp/cotizacion.js`

===============================================================================
7. FLUJO GENERAL DE DATOS
===============================================================================

1. el usuario abre un modulo HTML
2. `hub-config.js` resuelve backend, tenant y branding
3. `layout.js` protege sesion y permisos
4. `pb-core.js` crea cliente PocketBase con `tenant` activo
5. los modulos JS consultan colecciones (`espacios`, `cotizaciones`, etc.)
6. PocketBase aplica reglas de coleccion + hooks JS
7. los modulos generan `desglose_precios`, `espacios_detalle`,
   `detalles_evento` y documentos
8. PDFs y archivos quedan ligados a `cotizaciones` y/o `documentos`

Migraciones relevantes:

- `1700000001_init_cotizador.js`
  colecciones base
- `1773300100_fix_rules_native_tenants.js`
  reglas multi-tenant nativas
- `1780000007_lock_public_quotes_reads.js`
  cierra lectura publica de cotizaciones
- `1790000004_fix_app_users_create_rule.js`
  obliga alta de usuarios solo por admin
- `1790000005_espacios_permite_convenio.js`
  habilita convenio por espacio

11. DISPONIBILIDAD Y FECHAS
===============================================================================

Plaza Mayor:

- bloqueo por cotizaciones aprobadas/finalizadas
- permanencia mensual salvo configuracion personalizada

Casa de Piedra:

- disponibilidad construida desde `espacios_detalle`
- considera:
  - fechas del evento
  - `premontaje_fechas`
  - conceptos `b2b_montaje`
  - convenios con bloqueo indefinido

Funciones clave CP:

- `__cpBuildReservationsMap`
- `__cpEvalAvailability`
- `__cpQuoteBlocksIndefinitely`

===============================================================================
15. PUNTOS DE CUIDADO
===============================================================================

- `backend/pb_data/` es estado vivo, no debe editarse manualmente
- las migraciones deben ser compatibles con clon limpio
- cualquier cambio multi-tenant debe respetar `tenant`
- en Casa de Piedra no se debe mezclar `publicidad` con `espacio`
- cambios en convenio deben reflejarse tanto en `espacios_detalle` como en
  `detalles_evento.convenio`
- cualquier ajuste de PDF requiere validar preview, guardado y descarga
